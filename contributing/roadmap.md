# Terrain 开发方向 — Roadmap

> 参考 Claude Code release notes 的开发策略：**70%的工作不是加新功能，是让已有功能在所有场景下可靠。**

---

## 开发优先级栈

按优先级从高到低排列。**上层不稳固时，不开下层的新工作。**

```
P0  让现有功能不炸        修bug、修边界条件、修crash、修数据损坏
P1  让错误可诊断          改善错误信息、加日志、加诊断工具
P2  让更多场景跑得通      更多语言、更大代码库、更多OS、更多LLM后端
P3  让高级用户更爽        MCP工具增强、CLI体验、配置灵活性
P4  加真正的新功能        每个版本最多1-2个新功能
```

---

## P0：可靠性（持续进行）

**目标**：任何已暴露的MCP tool和CLI命令，在任何已支持语言的代码库上，都不应crash或返回错误数据。

| 方向 | 具体工作 | 理由 |
|------|---------|------|
| 大代码库韧性 | 10万+函数的代码库索引不OOM、不超时 | Claude Code在这里修了大量内存泄漏 |
| 增量更新可靠性 | git-diff增量索引在rebase、force-push、branch切换后仍然正确 | 这是最容易出边界条件的地方 |
| 数据库并发安全 | Kùzu在MCP server多tool并发调用时不死锁 | MCP client可能并发调用多个tool |
| Embedding中断恢复 | checkpoint机制在各种中断场景下都能正确恢复 | 已有checkpoint，需要覆盖更多边界 |
| 跨平台路径处理 | Windows路径、空格路径、非ASCII路径全部正确 | Claude Code 30%的bug修复是路径相关 |

**每日实践**：每次改代码前，先跑 `pytest tests/ -v`。每次修bug，先写复现测试。

---

## P1：可诊断性

**目标**：当功能失败时，用户（或AI agent）能从错误信息中自行诊断问题。

| 方向 | 具体工作 | 理由 |
|------|---------|------|
| MCP工具错误信息 | 每个MCP tool返回的错误必须包含：什么失败了、为什么、怎么修 | Claude Code花了大量版本改善错误信息 |
| 索引状态诊断 | `terrain info` 输出每个pipeline步骤的完成状态和最后运行时间 | 类似 `claude doctor` 的诊断能力 |
| 日志分级 | 正常运行零日志；`CGB_DEBUG=1` 时输出完整trace | 类似Claude Code的OTEL tracing策略 |
| 失败恢复指引 | 索引中断后重新运行时，自动提示「检测到上次未完成，从步骤X恢复」 | 用户不应该需要猜测系统状态 |

## P2：场景覆盖

**目标**：让Terrain能索引更多类型的代码库，在更多环境下运行。

### 语言支持完善

| 语言 | 当前状态 | 目标 | 优先级 |
|------|---------|------|--------|
| Go | 40%（类型声明） | 完整支持（方法、接口、goroutine调用） | 高 — 用户需求大 |
| C/C++ 函数指针 | 30%（query设计完成） | 间接调用检测和解析 | 高 — 系统代码准确性 |
| Scala / C# / PHP | 基础支持 | 完善调用解析和类型推断 | 低 — 按需推进 |

### 规模适配

| 场景 | 目标 |
|------|------|
| 超大monorepo（>50万行） | 分片索引、按目录增量 |
| 多语言混合项目 | 跨语言调用链追踪（如Python调C扩展） |

### 环境适配

| 场景 | 目标 |
|------|------|
| Docker容器内运行 | 无GUI依赖、最小镜像 |
| CI/CD集成 | `terrain index` 作为CI步骤，输出可缓存 |
| 多LLM后端 | 支持OpenAI / Anthropic / 本地模型切换 |

---

## P3：体验增强

**目标**：让已有功能用起来更顺畅，不增加概念复杂度。

| 方向 | 具体工作 | 对标Claude Code |
|------|---------|----------------|
| MCP tool返回格式优化 | API doc返回self-contained markdown，agent不需要二次调用 | Claude Code的focus mode：一次调用返回完整信息 |
| 索引进度反馈 | `terrain index` 显示实时进度条和ETA | Claude Code的rate-limit重试信息改善 |
| 配置热重载 | 修改.env后不需要重启MCP server | 已有 `terrain reload`，确保MCP侧也生效 |
| 仓库自动发现 | MCP server启动时自动检测当前git仓库并索引 | 减少setup步骤 |
| API doc缓存 | L3文档生成后缓存，避免重复LLM调用 | 成本控制 |

## P4：新功能

**原则**：每个版本最多加1-2个新功能。新功能必须满足「Before You Start」六条原则。

### 近期候选（按价值排序）

| 功能 | 描述 | 为什么值得做 |
|------|------|------------|
| **变更影响分析** | 给定一个函数，分析修改它会影响哪些调用链和下游功能 | 这是agent最需要的「blast radius」能力 |
| **代码演进时间线** | 结合git history，展示函数的修改历史和调用关系变化 | 帮助理解「为什么代码是现在这样」 |
| **跨仓库搜索** | 同时搜索多个已索引仓库 | 微服务架构的核心需求 |
| **自然语言代码导航** | 「找到处理用户认证的代码」→ 返回相关函数和调用链 | find_api的进化版，结合RAG |
| **API变更检测** | 对比两次索引的diff，报告新增/删除/修改的公共API | 类似breaking change检测 |

### 远期方向

| 方向 | 描述 |
|------|------|
| **实时索引** | 文件保存时自动更新图谱，不需要手动 `terrain index` |
| **团队协作** | 共享索引服务器，多人共用一份图谱 |
| **IDE集成** | VSCode/JetBrains插件，在编辑器内展示调用关系 |

## 每日开发节奏

参考Claude Code的发布节奏（每天1个版本，每版10-40个条目），Terrain的节奏建议：

```
每天的工作分配：
├── 50% — P0/P1：修一个bug或改善一条错误信息
├── 30% — P2/P3：完善一个语言支持或优化一个交互细节
└── 20% — P4：推进一个新功能的一个切片
```

**关键原则**：
1. 每个PR只做一件事
2. 每个bug修复带一个回归测试
3. 新功能先写50行验证脚本，再集成到框架里
4. changelog条目用用户能理解的语言写，不用内部术语

## 衡量标准

不用功能数量衡量进度，用这些指标：

| 指标 | 怎么看 |
|------|--------|
| **测试通过率** | `pytest tests/ -v` 全绿 |
| **dep_check** | 零违规 |
| **边界条件覆盖** | 每个MCP tool至少有一个「翻车场景」的测试 |
| **错误信息质量** | 每个错误返回包含 what/why/how-to-fix |
| **索引成功率** | 对已支持语言的开源项目索引成功率 > 99% |
