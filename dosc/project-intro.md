# Code Graph Builder：面向 Agent 的代码仓库知识基础设施

## 项目定位

**Code Graph Builder（CGB）** 是一套专为 AI Agent 设计的代码仓库知识基础设施。

当前主流 AI 编程助手（如 Claude Code、Cursor）在面对大型陌生代码仓库时，往往面临一个根本性问题：**它们对代码库的理解是碎片化、非结构化的**。Agent 只能逐文件阅读，无法感知跨文件的调用关系、模块结构和隐藏的语义联系，由此产生大量幻觉——凭空推测不存在的函数、调用关系、或者模块路径。

CGB 解决的正是这个问题。它以代码仓库为输入，自动提取代码中的**所有显性结构**（函数定义、调用链、继承关系、模块层级）与**隐性语义**（函数意图、设计模式、功能边界），构建出一套完整的知识图谱，并通过标准化接口直接向 Agent 提供结构化的精准知识，让 Agent 在任何规模的代码库上都能"有据可查"。

---

## 核心问题与解决思路

### 问题：Agent 在大型代码库中的三大困境

| 困境 | 表现 | 根因 |
|------|------|------|
| **结构盲区** | 不知道函数在哪、谁调用谁 | 缺乏跨文件的结构化索引 |
| **语义缺失** | 无法理解模块意图和设计边界 | 代码本身不携带语义摘要 |
| **上下文爆炸** | 注入大量原始代码导致 context 溢出 | 无信息压缩和精准检索机制 |

### 解法：四步流水线，将代码仓库转化为 Agent 可直接消费的知识

```
源代码仓库
    │
    ▼
[Step 1] AST 解析 & 知识图谱构建
    树结构 + 调用链 + 继承关系 → Kuzu 图数据库
    │
    ▼
[Step 2] API 文档生成
    图谱数据 → 三层 Markdown 文档（项目级 / 模块级 / 函数级）
    │
    ▼
[Step 3] 向量嵌入
    函数文档 → 1536 维向量索引
    │
    ▼
[Step 4] Wiki 生成
    聚类 + LLM → 多页语义 Wiki
    │
    ▼
MCP 标准接口
    └─ Claude Code / Cursor / 任意 MCP 客户端
```

每一步的输出都直接服务于 Agent 的不同查询场景：精确定位、语义搜索、关系追踪、整体理解。

## 技术实现

### 知识图谱构建（Step 1）

使用 **Tree-sitter** 对源代码进行 AST 解析，当前支持 12 种主流编程语言：

> C / C++ · Python · JavaScript · TypeScript · Rust · Go · Java · Scala · C# · PHP · Lua

对每个代码文件，系统并行执行四类提取：

- **定义提取**：函数、类、方法、接口、枚举，含签名、可见性、行号
- **调用关系**：基于 Trie 索引的跨文件调用解析，将 `foo()` 映射到 `project.module.ClassName.foo`
- **导入关系**：跨模块依赖链追踪
- **结构层级**：Project → Package → Folder → File → Module 五层体系

所有提取结果持久化到 **Kuzu 嵌入式图数据库**（无需 Docker，开箱即用），支持 Cypher 图查询。一个中等规模仓库（约 3000 个函数）会产生超过 5000 条调用边。

### API 文档生成（Step 2）

将图谱数据转换为三层 Markdown 文档：

```
api_docs/
├── index.md          # L1：全项目模块索引（模块名 | 函数数 | 类数 | 摘要）
├── modules/          # L2：每个模块的函数列表和类型定义
│   └── auth.md
└── funcs/            # L3：每个函数的完整文档（向量嵌入单位）
    └── auth.authenticate.md
```

L3 函数文档示例结构：

```markdown
# authenticate

> 验证用户身份，成功返回访问令牌，失败抛出 AuthError。

- 签名: `authenticate(user: str, password: str) -> Token`
- 返回值: Token
- 可见性: public
- 位置: src/auth.py:45-78
- 模块: auth

## Call Tree
authenticate
  └─ db.query_user
       └─ cache.get_session

## Called by (3)
- api.login_endpoint
- tests.test_auth.test_login
- cli.interactive_login

## Implementation
```python
def authenticate(user: str, password: str) -> Token:
    ...
```
```

对无文档注释的函数，系统会调用 LLM 自动生成简洁的功能描述，补全语义空白。

### 向量嵌入（Step 3）

以 L3 Markdown 文档为单位，通过 **Qwen3 Embedding**（1536 维）生成向量索引，支持自然语言语义搜索。

### Wiki 生成（Step 4）

基于向量相似度对函数进行语义聚类，LLM 为每个功能簇生成一页 Wiki，涵盖：模块概述、核心函数、调用关系可视化、使用示例。输出为人类可读、Agent 可直接引用的 Markdown 文档。

## 关键创新：API 文档采用 Markdown 而非 JSON

这是 CGB 区别于同类系统的一个重要设计决策。

传统代码分析工具（如 LSP 协议、各类代码索引器）普遍以 JSON 格式输出结构化数据。CGB 选择 **Markdown 作为主要的知识表示格式**，原因如下：

### Token 效率对比

传递同一个函数的元信息，两种格式的开销差异显著：

```json
// JSON：结构符号消耗大量 token
{
  "name": "authenticate",
  "params": [{"name": "user", "type": "str"}, {"name": "password", "type": "str"}],
  "return": "Token",
  "calls": ["db.query_user", "cache.get_session"]
}
```

```markdown
<!-- Markdown：同等信息量，token 更少，更接近自然语言 -->
- **authenticate**(user: str, password: str) → Token
- 调用: db.query_user → cache.get_session
```

在大型代码库场景下（数千个函数的上下文），这一差异直接决定 context window 的利用率，进而影响 Agent 单次查询所能覆盖的知识密度。

### LLM 的"原生亲和度"

| 格式 | 训练语料来源 | LLM 表现 |
|------|-------------|---------|
| **Markdown** | GitHub README、技术文档、Wiki、博客 | 极强，接近"母语"；标题层级天然形成语义分块 |
| **JSON** | API 响应、配置文件、数据集 | 较强，但深层嵌套（>3 层）时易丢失层级关系；长数组中间元素易被忽略（lost-in-the-middle 效应） |

### JSON 的已知缺陷（在 Agent 上下文注入场景）

- 深层嵌套时 LLM 容易混淆层级关系
- 长数组中间段内容容易被"跳读"或遗漏
- 严格语法约束导致 LLM 生成时频繁出现格式错误（漏逗号、多括号）
- 大段 JSON 作为 context 注入时，LLM 倾向于对结构符号脱敏，信息提取率下降

### CGB 的格式策略

```
知识存储 / Agent 上下文注入  →  Markdown（L1/L2/L3 文档、Wiki）
工具调用 / 结构化交互        →  JSON（MCP 协议层、程序间通信）
混合场景                    →  Markdown 包裹 JSON 片段
```

这一设计使 Agent 在查询代码知识时，获得的是 LLM 最易理解的格式，而非需要二次解析的结构化数据，从根本上降低了幻觉产生的概率。

## 对 Agent 减幻觉的直接效果

CGB 从以下三个维度系统性地压缩 Agent 的幻觉空间：

**1. 精确定位取代猜测**
Agent 可以通过 `locate_function` 或 `find_api` 直接获取函数的确切文件路径和行号，不再需要推测"这个函数应该在哪里"。

**2. 调用图取代推断**
`Call Tree` 和 `Called by` 字段将函数的上下游关系明确呈现，Agent 不需要通过阅读大量源码来推断调用链。

**3. 语义检索取代全文扫描**
向量语义搜索允许 Agent 用自然语言（"解析算术表达式的函数"）精准找到目标，而非逐文件暴力匹配。

## 系统接口

CGB 通过 **MCP（Model Context Protocol）** 标准协议对外暴露 19 个工具，覆盖全部使用场景：

| 类别 | 工具 | 说明 |
|------|------|------|
| **仓库管理** | `initialize_repository` | 一键执行完整四步流水线 |
| | `switch_repository` | 切换已索引仓库 |
| **代码搜索** | `find_api` | 主搜索入口：语义搜索 + API 文档融合 |
| | `semantic_search` | 纯向量相似度搜索 |
| | `query_code_graph` | 自然语言 → Cypher 图查询 |
| | `get_code_snippet` | 按全限定名获取源代码 |
| | `locate_function` | 精确定位函数行号 |
| **API 文档** | `list_api_docs` | 浏览 L1/L2 层文档 |
| | `get_api_doc` | 读取 L3 函数详情 |
| **Wiki** | `list_wiki_pages` | 列出所有 Wiki 页面 |
| | `get_wiki_page` | 读取指定 Wiki 内容 |
| **代码指导** | `prepare_guidance` | 依据设计文档生成代码实施建议 |

工具支持实时进度回调，长时间任务（如建图、嵌入）在 MCP 客户端侧显示进度条。

## LLM 兼容性

系统支持四类 LLM 后端，按优先级自动检测，无需手动切换：

```
1. 通用 LLM_API_KEY / LLM_BASE_URL（最高优先级）
2. LiteLLM 代理（支持 100+ 提供商）
3. OpenAI 兼容 API（DeepSeek、本地模型等）
4. Moonshot/Kimi（默认兜底）
```

LLM 仅用于三类生成任务（函数描述生成、Wiki 生成、自然语言转 Cypher），不参与图构建和向量嵌入的核心流程，因此 LLM 质量对知识图谱的准确性影响有限。

## 总结

CGB 的核心价值在于：它不是一个"让 Agent 读更多代码"的工具，而是一套**让 Agent 不再需要猜测**的基础设施。通过将代码仓库的全部显性结构与隐性语义预先提取、索引、并以 LLM 最友好的 Markdown 格式呈现，CGB 将 Agent 在代码任务上的幻觉风险从根源上系统性地压缩，使其具备真正可靠的大型代码库操作能力。

## 技术深度剖析

### 一、编译器前端的核心挑战：从源码到符号的精确映射

构建代码知识图谱本质上是在做编译器前端的工作——但要应对比编译器更苛刻的条件：不能崩溃、不能报错、必须对所有语言的所有边界情况保持稳健。

#### 1.1 多语言 AST 解析的统一化

系统选用 **Tree-sitter** 作为 AST 解析引擎，而非为每种语言分别实现解析器。Tree-sitter 是一个增量式解析框架，核心优势在于：为所有支持的语言提供统一的查询接口（S-expression 查询语言），只需为每种语言编写声明式查询规则，而无需手动遍历 AST 树。

这带来了一个关键的架构决策：**语言差异通过配置（LanguageSpec）而非代码分支来处理**。每种语言的函数节点类型、类节点类型、调用表达式类型等，都被抽象成参数化的配置项。同一套处理管道在不同语言上运行，差异只存在于"查询什么"，而非"怎么查询"。

这种设计使得新语言的接入成本极低——只需定义对应的 LanguageSpec 和 Tree-sitter 查询，而无需修改核心处理逻辑。

#### 1.2 完全限定名（FQN）的构建

所有提取出的符号都需要被赋予一个全局唯一的完全限定名（Fully Qualified Name），这是整个知识图谱的主键体系。FQN 的构建规则为：

```
project.module.submodule.ClassName.method_name
```

看似简单，实际处理中充满边界情况：
- **包文件的识别**：Python 的 `__init__.py`、Rust 的 `mod.rs` 需要识别为包而非模块，否则会产生错误的层级关系
- **嵌套类和方法**：类内的方法需要追溯父节点层级才能正确构建 `Module.Class.method` 的路径
- **C 语言头文件**：同一函数可能在头文件中声明、在实现文件中定义，需要跨文件关联

#### 1.3 C 语言的特殊挑战

C 语言的代码结构在所有支持语言中最复杂，因此有专门的处理逻辑：

**可见性推理**：C 没有 `public`/`private` 关键字，可见性必须通过以下三个信号来推断：
1. 是否带有 `static` 关键字（文件内部可见）
2. 是否在头文件中有对应声明（跨模块公开）
3. 是否被其他文件导入

这要求文件处理必须有**明确的顺序**：头文件（`.h`）总是优先于实现文件处理，这样在处理 `.c` 文件时，已经知道哪些函数在头文件中被声明为公开接口。

**注释提取**：C 函数的文档注释（包括 Doxygen 风格的 `/***/`）位于函数定义的相邻上方。系统通过向上扫描 AST 兄弟节点来收集注释块，并处理注释与函数之间的空行中断——如果注释与函数相隔超过一行，则认为注释属于别处。

### 二、函数调用解析：从模糊引用到精确边

知识图谱中最有价值的信息是**调用关系**（CALLS 边），但也是最难精确提取的部分。在源码中看到的调用往往是不完整的引用：

```c
result = parse_btype(ctx);
```

系统需要将 `parse_btype` 解析为 `tinycc.tccgen.parse_btype`，才能在图中建立精确的边。这个解析过程面临以下挑战：

#### 2.1 四级渐进式消歧策略

系统的调用解析器（CallResolver）采用优先级递减的四级策略：

**第一级：自调用解析**
处理 `self.method()` / `this.method()` 形式，直接在当前类上下文中查找方法，准确率接近 100%。

**第二级：限定调用解析**
处理 `module.func()` 形式，先查 import 映射表将首部替换为完整路径，再验证注册表中是否存在。例如 `db.query_user()` 中的 `db` 可能通过 `import myapp.db as db` 引入。

**第三级：导入表直查**
如果 `func_name` 本身就在 import 映射中（`from auth import authenticate`），直接返回导入的全限定名。

**第四级：注册表后缀匹配**
以上三级都失败时，在整个函数注册表中做后缀匹配：找到所有以 `.parse_btype` 结尾的 FQN。这是最后的手段——当只有唯一匹配时准确，当存在同名函数时可能产生歧义。

这种渐进式策略的核心思想是：**在保证可靠性的条件下最大化覆盖率**。宁可某些边不建立（无法解析），也不引入错误的边（错误解析）。

#### 2.2 Trie 树加速符号查找

函数注册表采用 **Trie 树 + 哈希表**的双索引结构：
- Trie 树支持前缀搜索（找出 `project.module` 下的所有函数）和后缀匹配（找到所有名为 `parse_btype` 的函数）
- 哈希表支持 O(1) 的精确查找（验证某个 FQN 是否存在）

在大型仓库中（数千个函数），单纯的线性扫描后缀匹配代价过高。Trie 通过共享路径前缀大幅压缩内存，同时提供比线性扫描更高效的遍历能力。

#### 2.3 类型辅助解析

对于 `obj.method()` 形式的调用，需要知道 `obj` 的类型才能确定 `method` 属于哪个类。系统实现了一个**轻量级类型推理引擎**，不追求完整的静态类型分析，而是聚焦于最常见的几种场景：

- 变量声明中有显式类型注解（`User user = ...`）
- 变量赋值为已知类型的字面量（字符串、整数、列表）
- 变量通过 import 引入（`from models import User; u = User()`）

这种"够用就好"的设计哲学贯穿了整个系统——在不增加大量复杂度的前提下，覆盖实际代码中 80% 的常见模式。

### 三、多阶段流水线的协调与缓存

系统的四步流水线（建图 → API 文档 → 向量嵌入 → Wiki 生成）有严格的依赖关系：每一步依赖上一步的输出。为了避免在开发和重建场景中重复执行耗时操作，系统设计了**分层缓存机制**：

| 缓存层 | 存储形式 | 重建代价 | 失效条件 |
|------|---------|---------|---------|
| 知识图谱 | Kuzu 嵌入式数据库 | 高（5-15 分钟，需解析所有文件） | 源码变更 |
| 向量索引 | Pickle 文件 | 中（需 API 调用，1-5 分钟） | 嵌入模型变更 |
| Wiki 结构 | Pickle 文件 | 低（LLM 结构规划，约 30 秒） | 手动触发 |
| API 文档 | Markdown 文件目录 | 低（确定性生成，约 10 秒） | 图谱变更 |

这种分层设计允许用户在需要时只重建特定层级，而不必从头重跑全部流程。

#### 3.1 AST 缓存与内存管理

在 PASS 2（定义提取）和 PASS 3（调用提取）之间，系统需要重用已解析的 AST 树。一次解析一个大型 C 文件的 AST 代价较高，因此将 AST 结果缓存在内存中供 PASS 3 复用。

但大型仓库（数千个文件）的全量 AST 缓存可能占用大量内存。系统实现了一个**双重限制的 LRU 缓存**：
- **条目数限制**：超过上限时驱逐最旧的条目
- **内存限制**：超过内存阈值时批量驱逐（一次清除最旧的 10%，减少频繁驱逐的开销）

这是一个典型的实用主义设计——内存估算不精确（Python 的 `sys.getsizeof` 不包含引用对象），但足够防止 OOM，同时避免了引入复杂的精确内存跟踪机制。

#### 3.2 头文件优先的处理顺序

C/C++ 项目的文件处理顺序并非随意的，系统强制按以下规则排序：

1. 头文件（`.h`）总是在同名实现文件之前处理
2. 相同后缀的文件按路径字母顺序处理（确保构建结果的确定性）

这个顺序保证了头文件中的函数声明在实现文件被处理时已经在注册表中，使得可见性推理（判断某个函数是否有头文件声明）能够正确工作。

### 四、API 文档生成：为向量嵌入而设计

L3 函数文档不仅是人类可读的 API 参考，其首要设计目标是**成为高质量的向量嵌入单元**。这个目标决定了文档的结构方式：

#### 4.1 三层文档的职责分工

- **L1 索引**：供 Agent 了解仓库全貌，单次返回模块概览，token 消耗低
- **L2 模块文档**：供 Agent 快速浏览一个模块的所有接口
- **L3 函数文档**：向量嵌入的基本单元，也是 Agent 获取函数完整信息的最终来源

L3 文档的结构顺序经过刻意设计：**描述行（意图摘要）位于最前面**，是向量化时权重最高的部分。调用树、参数表、源代码等补充信息依次排列，为向量相似度计算提供上下文，也为 Agent 重读时提供完整细节。

#### 4.2 LLM 生成描述的批量策略

对于没有文档注释的函数（在 C 语言代码中极为常见），系统通过 LLM 自动生成一句话的功能描述。为了控制 API 成本，采用**批量生成**策略：每次将 10 个函数的签名和源代码片段打包发送给 LLM，一次调用返回 10 条描述。

批量策略带来了解析挑战——LLM 返回的是带编号的列表，需要通过正则表达式提取各条描述。系统支持多种常见的编号格式（`1.`、`1)`、`(1)`），以兼容不同 LLM 的输出风格。

#### 4.3 文件名安全与唯一性

L3 文档以函数的完全限定名作为文件名。但 C 函数的完整签名可能非常长（包含参数类型列表），超出文件系统的 255 字节限制。系统的处理方式是：

- 将 FQN 截断至 180 字节（UTF-8 安全截断，不截断多字节字符中间）
- 追加 8 位 MD5 哈希（基于完整 FQN）保证唯一性

这样既保持了文件名的可读性（主体部分仍是有意义的路径），又保证了唯一性和文件系统兼容性。

### 五、向量检索的工程细节

#### 5.1 查询与文档的嵌入不对称

代码检索中存在一个固有的语义不对称：**查询是自然语言，文档是代码**。直接用同一个嵌入模型处理两者，效果通常不理想，因为它们的语言风格差异悬殊。

Qwen3 Embedding 支持**任务特定指令**（Task Instruction）机制，允许在嵌入时附加任务描述，引导模型生成适合特定检索场景的向量表示。系统对查询端启用任务指令（"为代码检索任务编码此查询"），而对文档端不加指令。这种不对称嵌入策略有助于弥合查询语言与代码文档之间的语义距离。

#### 5.2 向量存储的两种模式

系统提供两种向量存储实现：

**内存模式（MemoryVectorStore）**：使用纯 Python 计算 cosine 相似度，将所有向量加载到内存中。适用于中小型仓库（数千个函数），无外部依赖，启动即用。

**Qdrant 模式**：接入专业向量数据库，支持持久化、过滤、大规模检索。适用于超大型仓库或多仓库共享索引的场景。

两种模式通过统一接口暴露，调用方无需感知底层实现。但两者在过滤逻辑上有一个值得注意的行为差异：内存模式是在检索后过滤结果，Qdrant 是在检索前过滤候选集。对于严格的精确过滤场景，这个差异可能影响召回率。

### 六、图数据库的设计选型：为何选择 Kuzu

市面上主流的图数据库（Neo4j、Memgraph）都需要独立运行数据库服务，这增加了部署复杂度（需要 Docker 或独立安装）。CGB 选择 **Kuzu** 作为默认图后端，核心理由是：

- **嵌入式**：Kuzu 直接以库的形式嵌入到 Python 进程中，无需独立服务，类似 SQLite 对关系数据库的定位
- **Cypher 兼容**：支持标准 Cypher 查询语言，与 Neo4j 生态兼容，不锁定用户
- **批量写入优化**：支持高效的批量节点/关系插入，适合一次性建图的场景

系统同时保留了对 Memgraph 的支持（通过相同的接口协议），用户可以根据需求切换到需要更强查询能力的独立数据库。

#### 6.1 Schema 预声明与关系表的多态处理

Kuzu 是 Schema-first 的图数据库，所有节点标签和关系类型需要在写入数据前通过 DDL 预先声明。这与 Neo4j 等支持动态 schema 的数据库不同。

最复杂的挑战来自**关系的多态性**：`CALLS` 这条边可能连接 `Function→Function`、`Method→Method`、甚至 `Method→Function`。Kuzu 的 REL TABLE 要求明确指定源节点和目标节点的标签类型，无法直接表达"任意→任意"的关系。

系统的处理方式是**运行时动态创建关系表**：首次遇到某个 label 对组合（如 `Method→Function`）时，检查是否已存在对应的关系表；如果不存在，自动创建 `CALLS_Method_Function` 表，并在覆盖映射表中记录这个映射。后续查询时，系统自动将对该关系类型的查询扩展到所有相关的物理表上。

#### 6.2 Cypher 值的安全序列化

将 Python 值安全地序列化为 Cypher 字面量需要处理多种类型：

- `None` → `NULL`
- `bool` → `true`/`false`（小写，Cypher 约定）
- `str` → 单引号包裹，内部单引号转义
- `list` → `[item1, item2, ...]`，每个元素递归序列化

这个序列化层防止了 Cypher 注入（类似 SQL 注入），同时处理了不同语言对 Python 基础类型的表示差异。

### 七、自然语言转 Cypher：让图数据库对 Agent 透明

`query_code_graph` 工具让 Agent 可以用自然语言提问（"哪些函数调用了 parse_btype？"），系统将其翻译成 Cypher 查询执行，Agent 获得结构化的图查询结果。

这个翻译过程通过 **LLM + 显式 Schema 提示**实现：将完整的图 Schema（11 种节点类型、10+ 种关系类型、各节点的属性字段）编码进系统提示，让 LLM 在充分了解数据结构的情况下生成查询。

几个关键的工程细节：

**温度设为 0**：Cypher 是严格语法的查询语言，任何格式错误都会导致查询失败。温度为 0 的确定性输出最大化了语法正确率。

**强制 LIMIT 50**：防止 Agent 意外触发全图扫描，保护性能和 context window。

**Markdown Fence 清理**：LLM 返回的 Cypher 通常被包裹在代码块（` ```cypher ` ）中，需要提取内部内容才能执行。系统做了后处理清理，兼容有无代码块两种情况。

### 八、整体系统的设计原则总结

回顾整个技术实现，有几条贯穿始终的设计原则：

**优雅降级，而非崩溃**
在每个可能出错的环节（文件解析失败、LLM 调用失败、调用无法解析），系统都选择记录日志并继续，而非抛出异常中断流程。一个有少量缺失边的知识图谱，远比一个因一个文件解析失败而无法构建的系统更有价值。

**实用主义的精度权衡**
类型推理不追求完整的静态分析，内存估算不追求精确计量，注释提取不追求完美的启发式规则。在每个维度，系统都选择了"覆盖 80% 的常见情况"而非"完美处理 100% 的情况"。这使得系统在合理的复杂度下具备了实用价值。

**参数化多样性，而非条件分支**
多语言支持通过 LanguageSpec 配置参数化实现，多 LLM 支持通过环境变量优先级实现，多向量存储支持通过接口抽象实现。这种设计使得扩展新语言、新 LLM 提供商、新存储后端的成本极低，不需要修改核心逻辑。

**分层缓存保护昂贵操作**
整个流水线的每一步都有独立的缓存层，昂贵的操作（AST 解析、LLM 调用、向量化 API）只在必要时执行。这使得系统在开发迭代和增量更新场景下具有合理的响应速度。
